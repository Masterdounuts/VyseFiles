# 2026-04-30 Memory

## Telegram Session Fix (2026-04-30 02:55 UTC)

**Problem:** Telegram messages created fresh sessions with no context from main webchat session.

**Root Cause:** No `session` config was set - default behavior is `per-peer` which isolates each sender.

**Solution:** Added to openclaw.json:
```json
"session": {
  "dmScope": "main"
}
```
This routes all Telegram DMs to the agent's main session (same as webchat), preserving context.

**Status:** Applied and gateway restarted. Test by sending via Telegram.

---

## Previous: Telegram Routing Discovery

**Issue:** User couldn't see Telegram messages in webchat

**Finding:** Messages were reaching but getting merged into webchat session. The channel metadata showed `webchat` even for Telegram-originated messages.

## Telegram Fix REVERSED (2026-04-30 03:32 UTC)

**Problem:** User said "I see the session that has telegram on it with replies I'm getting from you" - meaning Telegram replies were appearing in webchat, not as a separate Telegram session.

**Root Cause:** The `session.dmScope: "main"` setting was forcing ALL DMs (including Telegram) into the main webchat session.

**Solution:** Removed `session.dmScope` entirely from openclaw.json. This allows Telegram to create its own session.

**Status:** Gateway restarted. Telegram should now create a separate Telegram-specific session when you send a DM.

**Note:** This is the opposite fix from earlier - the original problem was "no context", the new problem was "wrong session routing".
