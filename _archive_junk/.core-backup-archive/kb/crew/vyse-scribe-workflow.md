# Vyse-Scribe Workflow (RON-Level)

*Updated 2026-04-27*

## The Rule
- Primary Brain: Control UI (~12 hrs active)
- Second Brain: GitHub (long-term, Scribe manages)

## CRITICAL: Scalable Features Stay in Primary
**If a feature is scalable, it stays in Control UI:**
- ✅ Skills (load on-demand)
- ✅ KB folders (load on-demand)  
- ✅ DREAMS.md (openclaw feature)

**Only move to second brain if:**
- Not scalable (static reference)
- Needs long-term archive

## Workflow

### When Vyse needs something from second brain:
1. Vyse asks Scribe: "fetch [info] from memory"
2. Scribe uses memory_search → retrieves data
3. Scribe reports back

### When second brain needs updating:
1. Vyse summarizes the info
2. Vyse gives to Scribe
3. Scribe commits & pushes to GitHub

### Before pushing to GitHub, ASK:
- "Is this scalable?" → If YES, keep in Control UI
- "Is this redundant?" → If YES, delete from GitHub

## Files Location
| Brain | Files |
|-------|-------|
| PRIMARY | HEARTBEAT, active, resume-point, skills/*, kb/quartermaster, kb/stocks, DREAMS |
| SECONDARY | memory/2026-04-*.md, kb/crew/*.md |

## Scribe's Job
- Keep GitHub organized
- Fetch when Vyse asks
- Commit changes after work
- NEVER let Vyse bypass to GitHub
- **ALWAYS check: is this scalable?**
